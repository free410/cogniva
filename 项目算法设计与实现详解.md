## 2. 算法总体目标

本项目本质上是一个面向知识问答场景的 RAG 系统，因此算法目标不是单一的“分类”或“生成”，而是由多个子目标组成：

1. 把原始文档转换为适合检索的知识单元。
2. 把知识单元映射到语义向量空间。
3. 在用户提问时，从知识库中召回最相关的 chunk。
4. 对召回候选进行二阶段重排，提高命中精度。
5. 为最终回答生成证据链和置信度报告。
6. 对高价值信息进行长期记忆调度。

因此，本项目使用的不是单一算法，而是一条“分块算法 + 向量编码算法 + 混合检索算法 + 重排算法 + 证据评估算法 + 间隔重复算法”组成的算法流水线。

---

## 3. 算法体系总览

### 3.1 算法模块一览

| 模块 | 主要算法/技术 | 作用 |
|---|---|---|
| 文档预处理 | 文本清洗、句子切分、段落切分 | 为后续分块与检索提供稳定输入 |
| 分块算法 | Recursive、Smart Hybrid、Intent-aware、Entity-aware、Deep Semantic | 把长文档切成适合检索的 chunk |
| 向量编码 | DashScope `text-embedding-v3` | 将 query 和 chunk 映射到 1536 维向量空间 |
| 稠密检索 | Cosine Similarity | 基于语义向量召回候选 chunk |
| 稀疏检索 | TF-IDF、BM25、术语精确匹配 | 基于关键词与术语进行字面召回 |
| 分数融合 | Dense + Sparse + Term Match 加权融合 | 得到 hybrid score |
| 重排算法 | CrossEncoder `BAAI/bge-reranker-v2-m3` + 规则重排 | 对候选结果进行二阶段精排 |
| 证据评估 | 一致性、覆盖度、可答性、粒度匹配、质量评分 | 计算 evidence report 与置信度 |
| 记忆调度 | SM-2 间隔重复算法 | 决定长期记忆的下次复习时间 |

### 3.2 端到端算法链路

从算法流程上看，本项目的主链路可以概括为：

`原始文档 -> 文本抽取 -> 智能分块 -> 向量化 -> 持久化 -> Query 向量化 -> Dense/Sparse 混合召回 -> 融合排序 -> CrossEncoder 重排 -> Prompt 构建 -> 证据生成 -> 回答输出`

## 4. 文档预处理与分块算法

文档分块是整个系统最重要的底层算法之一。因为 RAG 的召回对象不是整篇文档，而是文档切片。如果切片粒度不合理，后面再强的 embedding 和 rerank 都会被拖累。

### 4.1 预处理算法

`AdvancedChunker` 在分块前会先做文本预处理，主要包括：

- 规整空白字符
- 规整换行
- 句子级切分
- 段落级切分

代码中的典型方法包括：

- `_preprocess_text(text)`
- `_split_into_sentences(text)`
- `_split_into_paragraphs(text)`

其核心目标不是复杂 NLP，而是保证进入分块器的文本具有基础结构性。

### 4.2 Recursive Chunking 算法

递归切块是一种经典的基础策略。其核心思想是：

1. 先按粗粒度分隔符切分，例如多个换行。
2. 如果切完后块仍然过大，则继续按更细粒度分隔符切分。
3. 如果仍然过大，则退化为固定窗口切分。

代码实现中存在一个递归分隔符层次：

- `\n\n\n`
- `\n\n`
- `\n`
- 中文句末符号
- 英文句末符号
- 逗号
- 空格
- 字符级

这种策略的优点是稳定、简单、几乎适用于任何文本。缺点是它只利用结构边界，不理解语义边界。

### 4.3 Smart Hybrid Chunking 算法

项目默认推荐的策略是 `smart_hybrid`。它并不是单一规则，而是一个组合算法。基本流程如下：

1. 预处理文本。
2. 检测问答对结构。
3. 分析文档结构。
4. 句子级切分。
5. 检测语义边界。
6. 基于边界构建初始 chunk。
7. 对 chunk 做后处理，合并过小块、拆分过大块。

也就是说，`smart_hybrid` 的本质是：

- 结构信息参与边界判断
- 语义信息参与边界判断
- 问答对完整性参与边界判断
- 最后再用规则做工程修正

它比单纯的固定长度切分更适合知识问答场景。

### 4.4 Intent-aware Chunking 算法

`chunk_by_intent(text)` 的目标是避免把不同意图类型的内容混进同一个 chunk。该算法会尝试识别文本段落的意图类型，例如：

- 定义类
- 原因类
- 流程类
- 对比类

当出现以下情况时就倾向于开启新块：

- 当前块长度已接近目标大小
- 当前段落的意图与当前块已有意图冲突

这一策略适合“说明文、教程文、FAQ 文档”等带明显话语功能结构的内容。

### 4.5 Entity-aware Chunking 算法

`chunk_entity_aware(text)` 的核心思想是：尽量不要把同一命名实体相关的信息切散。

基本步骤如下：

1. 将文本切成句子。
2. 对每个句子提取简单实体集合。
3. 计算相邻句子的实体重叠情况。
4. 在实体连续性下降的位置设置边界。

这种算法的优势是：

- 适合实体密集型文档
- 有利于保留“对象 - 属性 - 关系”的局部完整性

其本质是“用实体连续性代替纯字符长度作为边界依据”。

### 4.6 Deep Semantic Chunking 算法

`chunk_deep_semantic(text, embedding_threshold=0.65)` 是项目中语义最强的一种分块方式。它的核心流程是：

1. 先将文本切成句子。
2. 为每个句子生成 embedding。
3. 计算相邻句子之间的余弦相似度。
4. 根据相似度变化检测语义断点。
5. 在断点处分块。

该算法使用动态阈值：

`dynamic_threshold = max(embedding_threshold, mean_sim - std_sim)`

其中：

- `embedding_threshold` 是基础下限阈值
- `mean_sim` 是相邻句子平均相似度
- `std_sim` 是相邻句子相似度标准差

这样做的原因是：不同文档内部的语义连贯性水平不同，固定阈值不一定适合所有文本，因此要根据文档内部统计特征做动态调节。

### 4.7 后处理算法

分块后，系统还会执行 `_post_process_chunks()`。它主要做两件事：

1. 合并过小 chunk
2. 拆分过大 chunk

其意义在于把“算法上合理但工程上不理想”的边界再修正成更适合检索的粒度。

常见策略包括：

- 如果 chunk 太小，尝试与相邻 chunk 合并
- 如果 chunk 太大，调用 `_split_large_chunk()` 再次拆分
- 如果超长块仍无法按句子结构平衡拆分，则退化为滑窗切分

### 4.8 Chunk 元数据算法

项目会为每个 chunk 生成结构化元数据，例如：

- `chunk_id`
- `chunk_index`
- `char_count`
- `token_count`
- `strategy`

其中 `token_count` 当前使用简化估计：

`token_count ≈ len(chunk) // 4`

这不是精确 tokenizer 计数，但对工程侧估计 prompt 成本已经足够。

## 5. 向量编码算法

### 5.1 使用的具体模型

项目当前主 embedding 服务是：

- 平台：阿里云 DashScope
- 模型：`text-embedding-v3`
- 输出维度：`1536`
- 返回格式：`float`

对应代码封装类为：

- `DashScopeEmbeddingService`

### 5.2 Embedding 的算法本质

Embedding 算法的目标是把文本 `x` 映射为高维向量 `e(x)`：

`e(x) in R^1536`

映射后，相似语义的文本在向量空间中会更接近。后续 dense retrieval 就是基于这一点进行的。

### 5.3 同步与异步生成

项目同时实现了两套接口：

- `embed(texts)`
- `embed_async(texts)`

这样文档入库和在线检索都可以根据上下文选择同步或异步方式调用。

### 5.4 缓存算法

`DashScopeEmbeddingService` 内部维护了简单缓存：

- key: 原始文本
- value: 对应 embedding

这是一种内容级缓存算法，作用是避免同一段文本重复调用远程 embedding API。

其适用前提是：

- 相同输入文本应得到相同输出向量
- 本地内存缓存规模在当前项目中是可接受的

### 5.5 降级算法

如果 DashScope 调用失败，项目不会直接中断，而是调用 `_fallback_embed(texts)` 生成“伪向量”。

该算法大致步骤如下：

1. 对文本做 `MD5`。
2. 取 hash 的前若干字节作为随机种子。
3. 生成随机高斯向量。
4. 对向量做 L2 归一化。
5. 按词数做轻微缩放。

这不是语义 embedding，只是一种工程降级方案。其目的不是保证检索质量，而是保证系统链路不断。

## 6. 稠密检索算法

### 6.1 余弦相似度算法

项目中 dense retrieval 的核心相似度函数是：

`cosine_similarity(vec1, vec2)`

计算公式为：

`cos(theta) = (v1 · v2) / (||v1|| * ||v2||)`

其中：

- `v1` 是 query embedding
- `v2` 是 chunk embedding

余弦相似度的优点是：

- 不依赖向量绝对长度
- 更关注方向一致性
- 非常适合 embedding 空间的语义相似比较

### 6.2 Dense Retrieval 主流程

在主检索过程中，系统会：

1. 为用户 query 生成 embedding。
2. 遍历候选 chunk 的 embedding。
3. 计算 query 与每个 chunk 的余弦相似度。
4. 记录每个 chunk 的最佳匹配 query 变体与相似度分数。

当前主路径是在 Python 内存中完成该计算，而不是直接依赖数据库做向量 ANN 搜索。

### 6.3 Query Expansion 现状

项目保留了 `_expand_query_with_hyde()` 作为查询扩展接口，但当前实现直接返回原始 query：

`return [query]`

这意味着：

- 架构上已预留 query expansion 位点
- 当前实现为了控制误召回，没有真正开启 HyDE 或生成式扩展

这是一个典型的“先保守、后增强”的工程策略。

## 7. 稀疏检索算法

稀疏检索的目标是补足 dense retrieval 的不足，特别是在以下情况下：

- 用户问题中出现具体类名、字段名、接口名
- 需要精确字面匹配
- 中文专业术语必须显式命中

### 7.1 TF-IDF 风格算法

项目中实现了一个轻量 TF-IDF 风格相关性函数 `_tfidf_score(query, content)`。

其本质是通过：

- 词频 `TF`
- 区分度 `IDF`

来衡量 query 中的词在文档块中的代表性。

TF-IDF 的核心思想是：

- 在当前文档块中出现越频繁的 query 词越重要
- 在全局或局部候选中越“稀有”的词区分能力越强

### 7.2 BM25 算法

项目中实现了 `bm25_score(query, document, k1=1.5, b=0.75)`。

BM25 是经典信息检索排序算法，可以看作 TF-IDF 的改进版。其基本思想是：

- query 词在文档中出现越多，相关性越高
- 但词频收益不是线性无限增长，而是边际递减
- 文档长度需要归一化，避免长文天然得分过高

标准 BM25 公式可以写为：

`score(D, Q) = sum( IDF(qi) * ( f(qi, D) * (k1 + 1) ) / ( f(qi, D) + k1 * (1 - b + b * |D| / avgdl ) ) )`

其中：

- `f(qi, D)` 是词 `qi` 在文档 `D` 中的词频
- `|D|` 是文档长度
- `avgdl` 是平均文档长度
- `k1` 控制词频饱和速度
- `b` 控制长度归一化程度

### 7.3 中文 BM25 的工程化处理

项目中的 BM25 实现不是直接照抄英文检索模板，而是做了中文适配：

- 支持中文与英文混合文本
- 尝试使用 `jieba`，如果环境中可用则启用
- 如果不可用，则退回到简化中文切分逻辑
- 维护了一批“重要术语”，避免被错误切碎

这类适配非常重要，因为中文文本没有天然空格分词，如果直接把英文 BM25 套到中文上，效果通常会明显下降。

### 7.4 术语精确匹配算法

项目除了 TF-IDF 和 BM25，还专门做了 `term_match`。基本思路是：

1. 先从 query 中提取核心术语。
2. 判断这些术语是否原样出现在 chunk 中。
3. 按命中比例计算 `term_ratio`。

这是一个非常适合工程知识库的算法，因为系统中的很多关键问法并不是抽象语义，而是精确符号：

- `RAGService`
- `document_ids`
- `pgvector`
- `CrossEncoder`

此时术语匹配往往比纯语义相似度更可靠。

### 7.5 短语匹配算法

项目中还实现了 `_calculate_phrase_match_score(query, content)`，用于检测查询中的重要短语是否在候选 chunk 中完整出现。

其价值在于：

- 避免只命中零碎关键词
- 优先保留真正覆盖原问题表达的 chunk

项目中短语匹配分数会直接参与后续 rerank 与最终排序。

## 8. 混合检索算法

### 8.1 算法动机

仅使用 dense retrieval 的问题是：

- 对精确术语不够敏感
- 对结构化字符串未必稳定

仅使用 sparse retrieval 的问题是：

- 对同义表达、改写表达、语义近似问题支持较弱

因此项目采用混合检索：

- dense 负责语义召回
- sparse 负责字面校正
- term/phrase 负责精确命中强化

### 8.2 分数归一化算法

因为 dense、BM25、TF-IDF 的原始数值范围不同，所以必须先归一化。

项目中的基本策略是：

- dense 分数按候选中的最大 dense 分数归一化
- sparse 分数按候选中的最大 sparse 分数归一化

归一化的目的，是让不同算法输出进入统一量纲。

### 8.3 当前主融合公式

项目当前融合逻辑有一个非常关键的分支：

当 `raw_bm25 > 0` 时：

`hybrid = bm25_norm * 0.4 + dense * 0.5 + term_match * 0.1`

当 `raw_bm25 == 0` 时：

`hybrid = dense * 0.8`

这说明当前算法有非常明确的设计意图：

1. 如果存在字面命中，则 BM25 应进入主导层。
2. 如果完全没有字面命中，则回退到语义相似度主导。
3. 术语匹配作为精确命中的小幅增强项。

### 8.4 候选过滤算法

在得到 `hybrid_score` 后，项目不会把所有 chunk 都送去重排，而是先做一轮候选过滤。当前过滤条件大致是：

- `bm25 > 0`
- `keyword > 0.0001`
- `term_match > 0`
- `dense_score > 0.3`
- `hybrid_score > 0.1`

满足这些条件之一的 chunk 才保留。

这是一个高召回、低误杀的策略：尽量保留潜在相关块，但过滤掉明显噪声。

### 8.5 候选截断算法

过滤后，系统会按 `hybrid_score` 降序排序，然后取：

`top_candidates = top_k * 10`

其原因是：

- 第一阶段需要多留一些候选供 rerank 精排
- 但又不能把候选集放得过大，否则 rerank 成本过高

## 9. 重排算法

### 9.1 使用的具体模型

项目当前使用的 CrossEncoder 模型为：

- 框架：`sentence-transformers`
- 类：`CrossEncoder`
- 模型名：`BAAI/bge-reranker-v2-m3`
- 设备：`cpu`

### 9.2 CrossEncoder 与 Embedding 的区别

Embedding 模型是：

- 分别编码 query 和 chunk
- 然后比较向量距离

CrossEncoder 是：

- 直接把 `(query, chunk)` 作为一对输入送入模型
- 模型直接输出相关性分数

因此，CrossEncoder 通常更准，但推理成本更高，适合作为二阶段 reranker。

### 9.3 `_cross_encoder_rerank()` 主精排算法

主检索流程里调用的是 `_cross_encoder_rerank()`。其核心步骤如下：

1. 构造候选集 `chunks_to_score`。
2. 对每个 chunk 计算 `phrase_score`。
3. 调用 `reranker.predict(pairs)` 得到原始重排分数。
4. 结合 `phrase_score`、`rank_score`、`original_score` 构造最终显示分数。

在当前实现中，若 `phrase_score > 0`，则 final score 会优先提升到更高分段；如果没有短语精确匹配，则分数更多依赖 reranker 排名与原始 hybrid score。

这说明项目并不是完全信任 CrossEncoder，而是把它与短语匹配和原始混合得分一起使用，形成更加保守的精排机制。

### 9.4 无 reranker 时的回退算法

如果 CrossEncoder 模型不可用，则项目退化为：

`final_score = phrase_score * 0.6 + original_score * 0.4`

这说明即使没有神经重排模型，系统仍然保留：

- 关键词/术语驱动的精确短语优先
- 原始 hybrid score 兜底

### 9.5 `_rerank_chunks()` 辅助重排算法

项目中还存在一个辅助重排函数 `_rerank_chunks()`，它将：

- 模型分数
- 规则分数
- 原始相似度

进一步融合为最终分数。

若 reranker 可用：

`final_score = original_score * 0.3 + reranker_score * 0.5 + rule_score * 0.2`

若 reranker 不可用：

`final_score = original_score + rule_score`

这是一种典型的“模型主导 + 规则补偿”算法。

### 9.6 规则重排算法

规则重排主要考虑以下因素：

1. 完整 query 是否直接出现在 chunk 中。
2. 核心术语出现频率。
3. 核心术语是否出现在文档开头附近。
4. 周围上下文是否含有相关提示词。
5. 文本长度是否合理。

具体加分逻辑包括：

- 完整查询命中：`+0.15`
- 术语频次加分：最多 `+0.25`
- 术语位置加分：`+0.10` 或 `+0.15`
- 上下文提示词加分：最多 `+0.10`
- 长度过短惩罚：`-0.10`
- 长度适中轻微加分：`+0.05`

其意义在于把纯神经模型不一定稳定表达的局部特征显式补进去。

## 10. 证据评估与置信度算法

本项目一个非常有代表性的算法能力，是不仅检索内容，还对“这次检索是否可信”做量化评估。

### 10.1 证据评估的八个维度

`build_evidence_report()` 中，系统会综合以下维度构建置信度：

1. 综合相关性
2. 向量相似度
3. 结果一致性
4. 术语匹配
5. 可答性
6. 查询覆盖度
7. 粒度匹配
8. 内容质量

### 10.2 权重设计

当前权重如下：

- `relevance_weight = 0.50`
- `vector_weight = 0.15`
- `consistency_weight = 0.15`
- `term_weight = 0.08`
- `answerability_weight = 0.05`
- `coverage_weight = 0.03`
- `granularity_weight = 0.01`
- `quality_weight = 0.01`

从这组权重可以看出系统的设计哲学：

- 第一优先级是“整体相关性”
- 第二优先级是“向量相关性 + 多证据一致性”
- 其余维度主要用于修正与解释

### 10.3 可答性算法

`_calculate_answerability_score(query, chunk_content)` 的目标是判断“这个 chunk 是否真的像一个答案”。

它会考虑：

- 问题类型匹配
- 答案模式词出现情况
- query 中实体是否被覆盖

也就是说，它不是只看相关，而是看该 chunk 是否具备“回答问题”的形态。

### 10.4 覆盖度算法

`_calculate_coverage_analysis(query, retrieved_chunks)` 会输出：

- `semantic_coverage`
- `lexical_coverage`
- `completeness`
- `diversity`

其核心思想是：

- 语义覆盖度：检索结果是否覆盖了查询的语义空间
- 词汇覆盖度：query 关键词是否被结果命中
- 完整性：结果是否足够完整
- 多样性：多个 chunk 是否互补而不是重复

### 10.5 一致性算法

`_calculate_consistency_score(retrieved_chunks)` 主要考察：

- top 结果是否足够突出
- 多个 chunk 的得分是否集中
- chunk 之间是否形成证据链

本质上它回答的问题是：

“这些结果是在共同支持同一个答案，还是只是碰巧各自沾点边？”

### 10.6 粒度匹配算法

`_calculate_granularity_match(query, chunk_content)` 评估的是：

- query 的信息需求是粗粒度还是细粒度
- 当前 chunk 的长度和内容密度是否匹配该需求

例如：

- “什么是 RAG” 需要较概括的 chunk
- “bm25_score 在代码里怎么参与混合打分” 需要较细粒度的 chunk

### 10.7 最终置信度计算

系统会先把各维度贡献加总为 `raw_confidence`，再根据：

- top similarity
- 检索结果数量

做轻度加成或惩罚，最后把置信度裁剪到 `[0.10, 0.98]` 区间。

这说明置信度并不是单纯由 top1 相似度决定，而是一个多因子综合评估值。

### 10.8 置信度等级映射

当前等级映射为：

- `>= 0.90`：极高
- `>= 0.75`：高
- `>= 0.55`：中
- `>= 0.35`：低
- `< 0.35`：极低

这套映射既用于前端显示，也用于辅助提示生成策略。

## 11. 上下文构建算法

检索到 chunk 后，系统需要把它们转为适合大模型消费的上下文。

### 11.1 基础上下文构建

`build_context(retrieved_chunks)` 会按顺序把 chunk 拼成：

- 来源标签
- 正文内容

并用分隔线连接。

### 11.2 增强上下文构建

`build_enhanced_context(retrieved_chunks, query)` 还会引入长度控制。当前最大长度约为：

- `max_length = 3000`

算法思路是：

1. 顺序加入高分 chunk。
2. 如果即将超长，则只截取剩余可容纳部分。
3. 保证最终 prompt 不超过预设上下文预算。

这是一种典型的“检索后预算约束”算法。

## 13. 长期记忆算法

项目的长期记忆部分使用的是：

- `SM-2` 风格间隔重复算法

### 13.1 记忆调度目标

记忆算法要解决的问题是：

- 什么内容值得复习
- 什么时候应该再次复习
- 记忆效果越好，复习间隔应越长
- 记忆效果越差，复习应越快回到眼前

### 13.2 当前基础间隔表

项目中的基础间隔近似为：

- 第 0 次：1 天
- 第 1 次：1 天
- 第 2 次：3 天
- 第 3 次：7 天
- 第 4 次：14 天
- 第 5 次：30 天
- 第 6 次：60 天
- 第 7 次：120 天

同时在高质量回忆时还会进一步延长到更远区间。

### 13.3 质量评分算法

复习质量 `quality` 的范围是：

- `0` 完全忘记
- `1` 几乎忘记
- `2` 记忆模糊
- `3` 基本记住
- `4` 记住
- `5` 完全记住

### 13.4 下次复习时间算法

`_calculate_next_review(importance, review_count, quality)` 的主要逻辑是：

- `quality = 0`：30 分钟后再出现
- `quality = 1`：1 天后
- `quality = 2`：2 天后
- `quality >= 3`：进入间隔重复序列

对于 `quality >= 3` 时：

`interval = base_interval * quality_factor * importance_factor`

其中：

- `quality_factor = 0.5 + (quality - 3) * 0.25`
- `importance_factor = 0.7 + importance * 0.6`

最后再把区间裁剪到 `[1, 365]` 天。

### 13.5 Easiness Factor 算法

项目还实现了 `SM-2` 中的 easiness factor 更新：

`new_ef = current_ef + (0.1 - (5 - quality) * (0.08 + (5 - quality) * 0.02))`

然后把 EF 限制在：

- 下限 `1.3`
- 上限 `2.5`

在项目中，`importance` 会被映射到 EF 的缩放结果，用于影响后续间隔长度。

### 13.6 关键词抽取算法

`_extract_keywords(text)` 当前采用轻量规则式方法：

- 用正则匹配连续 2 到 6 个中文字符
- 去掉简单停用词
- 取前 5 个关键词

这不是复杂关键词提取模型，但对于本项目已经足够支撑基础记忆标签生成。

### 13.7 对话记忆抽取算法

`extract_memories_from_conversation()` 会从最近若干条消息中挑出值得记忆的内容。判定规则主要包括：

- 是否是用户消息
- 是否包含问题
- 是否包含“重要、定义、规则、结论”等关键词
- 长度是否在合适范围内

这是一个启发式抽取算法，强调简单可运行。

## 14. pgvector 相关算法说明

虽然项目当前主检索路径主要在 Python 内存中完成，但代码中已经提供了 `_retrieve_with_pgvector()`。

### 14.1 pgvector 相似度检索公式

SQL 中使用的核心表达式是：

`1 - (v.embedding <=> :vector) as similarity`

其中 `<=>` 是 pgvector 的距离运算符。该表达式把“距离”转换为“相似度”。

### 14.2 当前状态

需要特别说明的是：

- 代码里已经有 pgvector 查询实现
- 但当前模型层 `Vector.embedding` 仍是 `JSON`
- 主检索路径仍主要依赖 Python 内存中的向量相似度计算

因此，pgvector 在当前项目中属于“已设计、未完全主用”的算法能力。

## 15. 算法复杂度与工程特征

### 15.1 Dense 检索复杂度

当前 Python 内存 dense retrieval 近似为：

- 时间复杂度：`O(N * d)`
- 其中 `N` 是候选 chunk 数量，`d` 是向量维度（1536）

这在中小规模知识库中可接受，但规模进一步扩大后会成为性能瓶颈。

### 15.2 CrossEncoder 重排复杂度

如果候选数为 `K`，则 CrossEncoder 推理近似为：

- 时间复杂度：`O(K)`

但每个样本推理成本很高，因此常见策略是：

- 先粗召回
- 再精排少量候选

项目当前正是这么做的。

### 15.3 Chunking 的复杂度

普通递归切块大致为线性复杂度。
深度语义切块则增加了句子 embedding 成本，因此其代价更高，但分块质量更好。

## 16. 当前算法实现的特点总结

本项目的算法实现具有以下几个明显特点：

1. 不是单一向量检索，而是完整的混合检索系统。
2. 不是只有召回，还包含精排、证据解释和置信度评估。
3. 不是只关注问答，还把长期记忆调度纳入算法体系。
4. 在算法设计上兼顾了“效果”和“能落地运行”。
5. 保留了 pgvector、query expansion 等扩展位点，说明后续还有继续优化空间。

---

## 17. 当前算法层面的不足

尽管当前算法体系已经较完整，但从算法工程角度仍有几个明显限制：

1. 向量检索尚未完全迁移到 pgvector 原生 ANN 路径。
2. Query expansion 位点已留，但当前未真正启用生成式扩展。
3. TF-IDF 与 BM25 更偏工程改良版，而非严格的标准大规模检索实现。
4. CrossEncoder 当前运行在 CPU 上，候选增多时延迟会增加。
5. 关键词抽取与记忆抽取仍主要依赖启发式规则。

## 18. 后续算法优化建议

### 18.1 检索层

- 将 `vectors.embedding` 从 JSON 迁移为真正的 `vector(1536)`
- 在 PostgreSQL 中建立 HNSW 或 IVF 索引
- 把 dense retrieval 从 Python 内存迁移到 pgvector SQL 查询

### 18.2 查询理解层

- 真正启用 query expansion
- 加入同义词扩展与术语别名库
- 加入多轮对话中的查询改写

### 18.3 重排层

- 将 CrossEncoder 从 CPU 迁移到 GPU 或推理服务
- 区分轻量重排和高精重排模式
- 引入更明确的学习型 rerank 策略

### 18.4 记忆层

- 用 LLM 总结式抽取替代纯规则抽取
- 引入知识去重和记忆合并算法
- 让记忆对回答策略产生更细粒度影响

## 19. 结论

从算法角度看，本项目已经不是简单的“上传文档然后向量检索”，而是一个具备较完整检索链路的中文知识问答系统。其算法体系已经覆盖：

- 智能分块
- 向量编码
- 稠密检索
- 稀疏检索
- 混合融合
- CrossEncoder 重排
- 证据与置信度评估
- 间隔重复记忆调度

尤其在检索方面，当前实现已经形成较完整的“两阶段混合检索”结构：

`Dense Retrieval + Sparse Retrieval -> Hybrid Fusion -> Candidate Filter -> CrossEncoder Rerank -> Evidence Report`

这说明项目在算法层面已经具有明确的系统性，而不是零散堆叠几个模型接口。
